Hardening reference

This page explains how to make your Linux server safer, step by step. Everything is written so even a total beginner can follow. Click any gray code box to copy the command. Before you run anything, replace the pink words with your real info.

01

Update your server (like updating your phone)

Why? Your server has software on it. That software can have holes that bad people use. When you “update,” you get the fixed versions. Do this first and do it often.

What to do: Run this one command on the server. It does two things: first it gets the list of latest stuff, then it installs the updates. The -y means “yes to everything” so it doesn’t stop to ask you.

sudo apt-get update && sudo apt-get upgrade -y

02

Change the SSH door (so robots can’t find it)

Why? By default, everyone connects to your server on “door” number 22. Bots on the internet constantly knock on door 22. If you use a different door (a different port), most of them never find you.

What to do: Pick a port number (e.g. 22022). Everywhere you see YOUR_PORT, type your number instead. Run the commands in order. The first two lines change the config file. The next block stops the old “door 22” and starts SSH on your new door. The last block tells the firewall: “allow my new door, block the old one.”

After this, you always connect with: ssh root@YOUR_SERVER_IP -p YOUR_PORT. YOUR_SERVER_IP = your server’s IP address.

sudo sed -i 's/^#*Port .*/Port YOUR_PORT/' /etc/ssh/sshd_config
grep -q "^Port " /etc/ssh/sshd_config || echo "Port YOUR_PORT" | sudo tee -a /etc/ssh/sshd_config
sudo systemctl stop ssh.socket
sudo systemctl disable ssh.socket
sudo systemctl enable ssh
sudo systemctl start ssh
sudo ufw allow YOUR_PORT/tcp
sudo ufw deny 22/tcp

To check it worked: run ss -tlnp | grep ssh on the server. You should see your port number, not 22.

03

Turn on the firewall (the bouncer)

Why? A firewall is like a bouncer: it only lets in traffic you said is OK. You already said “allow my SSH port” in step 2. Now we switch the bouncer on.

What to do: Make sure you’re connected with your new port (step 2) before you run this. Then run the command below. It will ask “may I turn on the firewall?” — type y and Enter.

sudo ufw enable

To see the rules: sudo ufw status

04

Only allow your computer to SSH (guest list)

Why? Right now, anyone on the internet can try to log in (they’ll hit the firewall and fail2ban, but we can do better). We tell the server: “only accept SSH from this one IP address — mine.” So even if someone knows your port, if they’re not from your IP, they’re not on the list.

Your “public” IP: It’s the address the internet sees when you go online from home (like 203.0.113.45). It is NOT 192.168.x.x — that’s only inside your house. To find your public IP, on your own PC run: curl -s ifconfig.me (or open whatismyip.com).

What to do: Replace YOUR_PUBLIC_IP with that number. Run the first command (add yourself to the allow list), then the second (block everyone else). Order matters: allow yourself first, then deny ALL.

echo "sshd: YOUR_PUBLIC_IP" | sudo tee -a /etc/hosts.allow
echo "sshd: ALL" | sudo tee -a /etc/hosts.deny

If your home IP changes later (many ISPs do that), run the first command again with the new IP or you’ll lock yourself out. Keep your host’s “web console” or “recovery” so you can fix it if that happens.

05

Install Fail2ban (auto-block people who guess passwords)

Why? Somebody can try thousands of passwords on your SSH. Fail2ban watches the logs: “This IP failed 5 times? Block it for an hour.” So bots and script-kiddies get banned automatically.

What to do: First install fail2ban. Then create a config file so it watches SSH on your port and doesn’t ban you. Replace YOUR_PORT with the same port from step 2, and YOUR_PUBLIC_IP with your IP. The ignoreip line means “never ban these” (your IP + localhost).

You can create the file with: sudo nano /etc/fail2ban/jail.local and paste the config, or use a single command to write it (replace the placeholders in that command). Then start fail2ban and check that the “sshd” jail is active.

sudo apt-get update && sudo apt-get install -y fail2ban

Contents of /etc/fail2ban/jail.local (replace YOUR_PORT and YOUR_PUBLIC_IP):

[DEFAULT]
bantime = 1h
findtime = 10m
maxretry = 5

[sshd]
enabled = true
port = YOUR_PORT
filter = sshd
logpath = /var/log/auth.log
maxretry = 5
ignoreip = 127.0.0.1/8 ::1 YOUR_PUBLIC_IP
sudo systemctl enable fail2ban
sudo systemctl restart fail2ban
sleep 3 && sudo fail2ban-client status sshd

06

Limit wrong passwords per try (MaxAuthTries)

Why? When you type the wrong password, the server lets you try again. If we say “only 3 wrong tries,” after the third wrong password it kicks that connection off. So one connection can’t try 100 passwords — they get 3 and then they’re out.

What to do: This sets the limit to 3. Run the first block (it adds or updates the line in the config), then restart SSH so it uses the new setting.

grep -q '^MaxAuthTries ' /etc/ssh/sshd_config && sudo sed -i 's/^MaxAuthTries .*/MaxAuthTries 3/' /etc/ssh/sshd_config || echo "MaxAuthTries 3" | sudo tee -a /etc/ssh/sshd_config
sudo systemctl restart ssh
Remember: YOUR_SERVER_IP = your server’s IP. YOUR_PORT = the SSH port you chose (e.g. 22022). YOUR_PUBLIC_IP = the IP of your home/office (from curl -s ifconfig.me). Always have a way to get in without SSH (your host’s web console or recovery) in case you lock yourself out.
